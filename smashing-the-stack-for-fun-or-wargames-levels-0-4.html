In the last days I have been playing the quintessential WarGames, a collection of Security problems, divided into 14 interesting titles. To play the WarGames you SSH to their servers with a login that indicates your current level. The purpose of the game is to solve the current level challenge to find the password for the next level.

Today I am talking about the first five levels of Narnia, which is all about buffer overflow and inputs with no bound checking. To solve the Narnia's challenges, you need to have some understanding of how a process's virtual memory works.

You will see that this war is not bad when you know your weapons.

A Crash Course of How Stack Exploitation Works

A Process' Virtual Memory

When a program starts a process, the OS kernel provides it a piece of physical memory. However, all that the process sees is the virtual memory space, with its total size and starting address. Each time a process wants to read or write to memory, its request must be translated from a virtual memory address to a physical memory address.

I like Peter Jay Salzman's picture to illustrate how the process's memory works:

The text and data segments are the places where the program puts the code to be executed and the static data (e.g, global variables). This region is normally marked read-only and any attempt to write to it will result in a segmentation violation. Notice that arguments and environment variables get a special place in the top of the stack (higher address).

The stack is a collection of stack frames which are the objective of this study. When a new function is called, a new frame needs to be added and the stack grows downward (lower memory address).

What's the Stack

A stack is an abstract data type that has the property that the last object placed will be the first object removed. This is also known as last in, first out queue, or a LIFO.Two of the most important operations in a stack are push and pop. You can think of a stack of books: the only way to reach the book in the bottom is popping every book until the first book that was pushed.

To learn more about how to write a stack in Python, for example, take a look into my notes on Python & Algorithms. I also made the source code available: here are some examples.

Every time a program calls a function, it will alters the flow of control. Once the function finishes its task, it will return to the same point where it was called. If you think about it, a stack is just the perfect object for this process: we push a function (ts arguments, code, etc) into the stack, then we pop it and we are back where we started. The stack dynamically allocates the local varibales in the functions, and return values in the function.

Buffer Overflows in the Stack

Buffer overflows happen when we give a buffer more information than it is meant to hold. The language C does not have any built-in bounds checking, so it allows overflow through its variables.

The standard C library has several functions for copying or appending strings with no bounds checking, for example: strcat(), strcpy(), sprintf(), vsprintf(), gets(), and scanf().

To understand better this subject, I recommend the classic from Aleph One's' Smashing the Stack for Fun or Profit, published in 1996 in the Phrack magazine number 49.

Assembly and the Stack Registers

First of all, a register called the stack pointer (SP) points to the top of the stack (lowest memory address) . The bottom of the stack (higher memory andress) is at a fixed address (adjusted by the kernel at run time). The CPU implements instructions to PUSH onto and POP off of the stack.

A frame pointer (FP) or local base pointer (LB) points to a fixed location within a frame.

When a process stats:

  1. The previous FP is saved so it can be restored at procedure exit).
  2. Then it copies SP into FP to create the new FP, and advances SP to reserve space for the local variables.
  3. Perfom all the operations
  4. In the end, return the saved return address.

In assembly: - 'call' will push the instruction pointer (IP) onto the stack. We'll call the saved IP the return address (RET).

    pushl %ebp
    movl %esp,%ebp
    subl $20,%esp
  • This pushes EBP, the frame pointer, onto the stack.
  • It then copies the current SP onto EBP, making it the new FP pointer.
  • We'll call the saved FP pointer SFP. It then allocates space for the local variables by subtracting their size from SP.

Memory can only be addressed in multiples of the word size, for example 4 bytes (or 32 bits).

Another way to look to the assembly output in a program is using -S switch to generate assembly code output:

$ gcc -S -o example1.s example1.c

To compile an assembly code we use:

$ nasm EXPLOITNAME.asm

It's a good skill to understand the basics of assembly. Check this nice Assembly guide if you need to.

Level 0: Warm-up with a classic Stack overflow to rewrite an variable.

The first challenge is in this code:

narnia0@melinda:/narnia$ cat narnia0.c

#include <stdio.h>
#include <stdlib.h>

int main(){
  long val=0x41414141;
  char buf[20];

  printf("Correct val's value from 0x41414141 -> 0xdeadbeef!\n");
  printf("Here is your chance: ");
  scanf("%24s",&buf);

  printf("buf: %s\n",buf);
  printf("val: 0x%08x\n",val);

  if(val==0xdeadbeef)
    system("/bin/sh");
  else {
    printf("WAY OFF!!!!\n");
    exit(1);
  }

  return 0;
}

The program receives an input from the user and save in a buffer variable of size 20. If the input variable is withing this range, nothing happens and the exit normally. However, what happens if the input is larger than 20 bytes? In this case, there is no bound checkings and the input will overflows the variable buf, occupung the subsquent space in the stack.

Now the magic, a Memory Stack, just like a pile of books, push the varibales in the owrder they are created in your prgram. In the case above, the first thing that the Stack got was val, then buf:

  long val=0x41414141;
  char buf[20];

This means that val is right below buf in the stack (in higher memory address), so if something overflows from buf will rewrite the value of val in the memory! This is a classical case of Stack Overflow!

The magic is that we can feel var with anything we want, and in the next levels we will see several different types of exploits using this logic. For now, being just the level 0, it's really simple: we need to make val have the hexadecimal value 0xdeadbeef:

if(val==0xdeadbeef)
        system("/bin/sh");

In this case, we see that the program will spawn a privileged (Bash) shell for us, where we can get access to the password for the next level.

  int i;
  char buffer[256];

  for(i = 0; environ[i] != NULL; i++)
